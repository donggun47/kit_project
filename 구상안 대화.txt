## 🏗️ 솔루션 명칭 (가칭): **Semantic Mind-Map Archive (SMMA)**
> **핵심 가치:** "파일 위치를 고민하는 '관리의 고통'을 없애고, 학습 데이터 간의 '맥락'을 이어주는 지능형 학습 비서"

### 1. 문제 정의 (Pain Point)
사용자가 언급한 **"위치 지정 및 찾기의 귀찮음"**을 교육 공학적 관점에서 재정의합니다.
* **파일 관리의 엔트로피:** 학습량이 많아질수록 폴더 구조가 복잡해지며, 결국 규칙이 무너지는 현상.
* **맥락의 단절:** 서로 연관된 과목(예: 수학과 물리)이 폴더라는 물리적 경계에 갇혀 지식의 연결이 어려움.
* **인출 실패 (Retrieval Failure):** 데이터는 저장되어 있으나, "어디에 두었는지" 기억나지 않아 학습 흐름이 끊김.

---

### 2. AI 솔루션 핵심 기능

#### ① 무지향성 데이터 저장 (Context-Aware Ingestion)
사용자가 파일명이나 위치를 정할 필요 없이 시스템에 "던지면", AI가 내용을 분석합니다.
* **기술적 구현:** RAG(Retrieval-Augmented Generation)를 위한 **Vector Database** 활용.
* **효과:** 비정형 데이터(노트, PDF, 이미지)를 텍스트로 임베딩하여 관계를 자동으로 형성합니다.

#### ② 다차원 검색 시스템 (Memory-Based Search)
정형화된 경로 검색이 아닌, **인간의 기억 메커니즘**을 모사한 검색을 지원합니다.
* **시맨틱 검색:** "어제 공부한 내용 중 미분 관련된 거"라고 입력하면 날짜 메타데이터와 내용 유사도를 결합해 즉시 제시.
* **관계형 시각화:** 파일 간의 연관성을 그래프 구조로 보여주어, 내가 공부한 지식들이 어떻게 연결되는지 한눈에 파악.


#### ③ 개인 맞춤형 AI 튜터 (Knowledge-Grounded Tutor)
단순한 챗봇이 아니라, **사용자가 직접 생성한 데이터**만을 기반으로 가르치는 선생님입니다.
* **맞춤형 질의응답:** 외부 정보가 아닌, "내가 정리한 노트"를 바탕으로 답변하여 학습 일관성 유지.
* **복습 자동화:** 저장된 데이터 간의 관계를 분석해 부족한 개념을 연결하여 질문을 던짐 (예: "지난번 배운 벡터 개념이 이번 물리 문제에 어떻게 적용될까?").

-------------------구조화---------------------------------


기존에 설계한 **Socratic Mirroring(거울 비추기)** 그래프만으로도 공부한 내용을 확인하는 것이 충분히 가능합니다. 시스템이 사용자에게 "맞다/틀리다"를 말하는 대신, **과거의 기록을 꺼내 현재의 사용자에게 질문을 던지는 루프**가 곧 '복습'이자 '확인' 과정이 되기 때문입니다.

그래프가 '확인 모드'에서 어떻게 작동하는지 검증 결과를 정리해 드립니다.

---

### 1. 복습/확인 시의 에이전트 작동 검증

사용자가 "나 쿼리에 대해 제대로 알고 있는지 확인하고 싶어"라고 요청할 경우의 흐름입니다.

| 단계 | 노드 이름 | 동작 (검증) | 비고 |
| :--- | :--- | :--- | :--- |
| **1단계** | **Recall_Node** | DB에서 '쿼리'와 관련된 사용자의 과거 정의, 메모, 연결된 개념들을 모두 추출합니다. | **사용자 데이터 기반** |
| **2단계** | **Mirroring_Node** | "사용자님은 이전에 쿼리를 '데이터베이스에서 명령을 내려 데이터를 조회, 삽입, 수정, 삭제하는 것'이라고 정의하셨습니다. 이 개념들이 각각 무엇인지 간단히 설명해 보시겠어요?" | **역질문(Active Recall)** |
| **3단계** | **Contrast_Node** | 사용자의 답변과 DB에 저장된 과거 정의를 대조하여 일치 여부를 판단합니다. | **AI 주도성 배제** |
| **4단계** | **분기 (Conflict)** | 답변이 과거 기록과 다르거나 부실할 경우 다시 질문(2단계)으로 돌아가거나, 원문 링크(`Archives`)를 제시합니다. | **자가 교정 유도** |

---

### 2. 그래프 흐름의 유연성 (학습 vs 확인)

동일한 그래프 구조가 목적에 따라 어떻게 쓰이는지 비교하면 구조적 완성도를 확인할 수 있습니다.

* **학습 시 (New Input)**: [신규 내용] → [기존 지식과 연결] → [새로운 통찰 저장]
* **확인 시 (Review)**: [기존 지식 소환] → [사용자의 현재 인출(Recall)] → [기존 지식과 대조/보정]



---

### 3. 구조적 보완점 (확인 모드 전용)

기존 그래프로 충분히 확인이 가능하지만, '테스트'의 느낌을 주고 싶다면 다음 **하나의 장치**만 추가하면 완벽합니다.

* **'빈칸 또는 힌트' 생성 로직**:
    사용자가 직접 정의한 문장에서 핵심 키워드만 가리고 "여기에 들어갈 개념이 무엇이었나요?"라고 묻는 로직입니다. 이는 AI가 새로운 정답을 만드는 것이 아니라, **사용자가 쓴 글을 문제지로 변환**하는 것이므로 사용자 주도권 원칙을 해치지 않습니다.

---

### 4. 최종 검증 결과: **[통과]**

추가적인 거대 그래프 설계 없이, 현재의 **[추출-소환-대조-질문]** 루프만으로도 충분히 학습 내용을 확인할 수 있습니다.

* **이유 1**: 사용자가 저장한 데이터 자체가 '정답지' 역할을 수행함.
* **이유 2**: `Mirroring_Node`가 사용자의 과거 발언을 근거로 반박하거나 질문하므로, 외부 정답 없이도 개념의 일관성을 체크할 수 있음.
* **이유 3**: 사용자가 막힐 때만 열리는 `External_Evidence` 창구가 최후의 검증 수단(보조 정답)이 됨.

---

이제 **기술 스택(FastAPI, PostgreSQL, LangGraph)**을 바탕으로 구현을 시작하셔도 좋을 만큼 논리적 구조가 탄탄해졌습니다. 시스템이 "당신은 100점입니다"라고 말하는 대신 **"당신의 과거 기록과 지금의 생각이 일치합니다"**라고 말해주는 진정한 의미의 아카이브 보조 AI가 될 것입니다.

구현 단계에서 특정 노드의 파이썬 코드가 필요하시거나, DB 쿼리 최적화가 필요하면 언제든 말씀해 주세요. 어떤 부분부터 구체화해 볼까요?


사용자의 철학을 유지하면서도 시스템이 '고립된 오류'에 빠지는 것을 방지하기 위한 **최소한의 외부 창구(External Gateway)**를 설계에 포함하겠습니다. 

AI가 먼저 외부 지식을 들이밀지 않되, 사용자가 자신의 논리에 확신이 없거나 벽에 부딪혔을 때만 **'반박용 외부 증거'**를 요청할 수 있는 기능을 추가합니다.

---

## 1. 수정된 랭그래프 흐름 (외부 창구 포함)

기존의 폐쇄적인 루프에 **'사용자 요청 기반 외부 검증'** 노드를 추가한 흐름입니다.

1.  **START**
2.  **분기1**: [DB 내용 기반 대조]
    * 내부 DB 및 과거 메모와 현재 발언 대조.
3.  **Mirroring_Node**: "과거 기록과 현재 내용이 다릅니다. 어떻게 생각하시나요?"
4.  **분기2 (사용자의 선택)**:
    * **A. 자가 교정**: 스스로 생각해서 답변 → **DB 저장(END)**.
    * **B. 외부 반박 요청**: **"내 논리가 맞는지 외부 자료로 검증해줘"**라고 요청.
5.  **External_Evidence_Node (신설)**: 
    * 웹 검색을 통해 사용자의 논리에 대한 **반박 자료(Counter-evidence)**나 **보충 자료**를 수집.
    * AI는 "이런 외부 자료가 있습니다. 여전히 같은 생각이신가요?"라고 다시 거울을 비춤.
6.  **Refinement & Commit**: 사용자가 외부 자료를 읽고 최종 정리 → **DB 저장(END)**.



---

## 2. 외부 창구 운영의 3대 원칙 (Constraint)

AI가 주도권을 뺏지 않도록 하기 위한 엄격한 가이드라인입니다.

* **요청 기반 작동 (On-Demand Only)**: 사용자가 명시적으로 "외부 자료 가져와", "반박해봐"라고 명령하기 전에는 절대 외부 검색을 수행하지 않습니다.
* **반박 중심 자료 제공 (Devil's Advocate)**: AI는 사용자를 칭찬하거나 동조하는 자료보다, 사용자의 논리를 **비판적으로 바라볼 수 있는 근거**를 우선적으로 찾아옵니다. 이는 사용자의 메타인지를 자극하기 위함입니다.
* **출처 투명성**: 외부 자료를 가져올 때는 요약본보다 **원문 링크와 핵심 문구**를 그대로 제시하여, 사용자가 직접 읽고 판단하게 합니다.

---

## 3. DB 설계 보완 (외부 데이터 기록)

외부 자료를 통해 지식이 수정되었을 경우, 그 계기를 기록하여 학습 경로를 추적합니다.

| 테이블명 | 추가 컬럼 | 용도 |
| :--- | :--- | :--- |
| **Concepts** | `external_ref_url` | 사용자가 수용한 외부 반박 자료의 링크 |
| **Activity_Logs** | `trigger_type` | 'Internal_Conflict'인지 'External_Challenge'인지 기록 |

## 4. 에이전트들의 예상 반응 (외부 반박 시나리오)

* **사용자**: "내가 쓴 쿼리 정의가 맞는 것 같은데? 외부 자료로 반박해봐."
* **External_Evidence_Node**: (웹 검색 후) "사용자님은 쿼리를 '찾는 것'이라고 하셨지만, 공식 문서나 기술 블로그(링크 첨부)에서는 '데이터를 정의(DDL)하거나 제어(DCL)하는 언어'도 쿼리의 범주에 넣고 있습니다. 이 부분에 대해 어떻게 생각하시나요?"
* **사용자**: "아, 명령(Command)의 개념이 더 넓구나. 정의를 수정할게."

| **PostgreSQL** | 사용자가 외부 자료를 수용한 '승인 로그'와 '수정된 정의'를 버전별로 보관. |

이렇게 하면 시스템은 기본적으로 사용자의 뇌를 쓰게 만들되, 필요할 때만 세상의 지식을 빌려와 **사용자의 논리를 단단하게 다져주는 도구**가 됩니다. 이 '선택적 외부 창구' 방식이 사용자님이 원하시는 균형점인가요?
